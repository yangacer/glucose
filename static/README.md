# Static Files Structure

This directory contains the frontend files for the Glucose Monitoring Dashboard.

## Directory Structure

```
static/
├── index.html              # Production HTML (auto-generated by build-js.py)
├── index.html.dev          # Development HTML (single source of truth for script order)
├── css/
│   └── styles.css          # All CSS styles
└── js/
    ├── config.js           # API base URL configuration
    ├── utils.js            # Utilities: API calls, timestamps, timezone, UI feedback
    ├── tabs.js             # Tab switching and navigation
    ├── dashboard.js        # Charts, summary timesheet, prediction
    ├── data-loader.js      # Data loading: nutrition/supplement options, autofill
    ├── dynamic-items.js    # Dynamic add/remove for intake form items
    ├── forms.js            # Form submission handlers with loading states
    ├── audit.js            # Audit listings, edit, and delete for all entities
    ├── main.js             # Application entry point
    └── release/
        └── app.min.js      # Minified production bundle (auto-generated by build-js.py)
```

## Development vs Production

**Development** (`MTLS_ENABLED=false`): server serves `index.html.dev`, which loads
individual JS files unminified — no build step needed, easy debugging.

**Production** (`MTLS_ENABLED=true`): server serves `index.html`, which loads the
single minified bundle `js/release/app.min.js`.

Build the bundle before deploying:
```bash
./build-js.py           # keep current version
./build-js.py 0.9.0     # bump version
```

`build-js.py` reads script order from `index.html.dev` (single source of truth),
combines files in dependency order, minifies with terser, and regenerates `index.html`.

## JavaScript Modules

### config.js
- `API_BASE` — API endpoint base URL; update here when server address changes

### utils.js

**Timestamp & timezone:**
- `getCurrentTimestamp()` — current time in `datetime-local` format (browser local)
- `toDbTimestamp(datetimeLocal)` — converts `datetime-local` → UTC string for server storage
- `toInputTimestamp(utcStr)` — converts UTC DB string → `datetime-local` format (browser local)
- `formatTimestamp(utcStr)` — converts UTC DB string → localised display string
- `getClientTz()` — returns browser IANA timezone via `Intl.DateTimeFormat`
- `resetToNow(inputId)` — resets a datetime-local input to current time
- `initializeDateInputs()` — sets default date ranges on page load

**API & UI:**
- `submitData(endpoint, data)` — generic POST handler
- `showMessage(elementId, success, message)` — shows success/error message with animations
- `setButtonLoading(button, loadingText)` — disables button and shows spinner
- `resetButton(button)` — restores button from loading state
- `escapeHtml(text)` — XSS-safe text escaping

### tabs.js
- `initializeTabs()` — sets up tab click handlers
- `switchTab(tabId)` — switches active tab
- `loadTabData(tabId)` — loads data when a tab is activated

### dashboard.js
- `loadDashboard()` — loads all dashboard components
- `loadGlucoseChart()` — renders glucose/insulin time-weighted mean chart
- `loadSummary()` — loads summary timesheet table
- `loadCVCharts()` — loads all three CV variability charts
- `loadRiskMetrics()` — loads all nine risk metric charts (LBGI/HBGI/ADRR)
- `loadPrediction()` — loads next-window glucose and insulin prediction
- `loadNutritionList()` — loads nutrition reference table
- `renderCVChart(canvasId, data, chartInstance)` — renders a single CV chart
- `renderRiskChart(canvasId, data, chartInstance, ...)` — renders a single risk chart
- `showSummaryOverlay(row)` / `hideSummaryOverlay()` — detail overlay for summary rows
- `getGlucoseColor(level)` — returns CSS colour class for a glucose value

All dashboard API calls append `&tz=${encodeURIComponent(getClientTz())}`.

### data-loader.js
- `loadNutritionOptions()` — loads nutrition dropdowns in intake form
- `loadSupplementOptions()` — loads supplement dropdowns in intake form
- `loadSupplementsList()` — loads supplements master table
- `autofillPreviousIntake()` — auto-fills intake form from previous 12-hour window
- Internal helpers: `populateNutritionSelect()`, `populateSupplementSelect()`,
  `populatePreviousNutritionItems()`, `populatePreviousSupplementItems()`

`autofillPreviousIntake()` calls `/api/intake/previous-window?tz=<clientTz>`.

### dynamic-items.js
- `addNutritionItem()` — adds a nutrition row to the intake form
- `addSupplementItem()` — adds a supplement row to the intake form
- `updateNutritionRemoveButtons()` — hides remove button when only one nutrition item remains
- `updateSupplementRemoveButtons()` — always shows remove (supplements are optional)
- `renumberNutritionItems()` / `renumberSupplementItems()` — updates labels after removal
- `initializeDynamicItems()` — attaches initial event listeners

### forms.js
- `initializeForms()` — sets up all form submit handlers
- `handleIntakeSubmit(e)` — handles multi-item intake form submission
- `resetIntakeForm()` — resets intake form to initial state

All form handlers use `setButtonLoading()` / `resetButton()` to prevent
double-submission and provide visual feedback during API calls.

### audit.js
Audit listings, inline editing, and deletion for all entities. Timestamps are
displayed in browser local time via `formatTimestamp()`. Edit prompts pre-fill
with local time via `toInputTimestamp()` and convert user input back to UTC via
`toDbTimestamp()` before submission.

All audit API calls append `&tz=${encodeURIComponent(getClientTz())}`.

- **Glucose:** `loadGlucoseAudit()`, `editGlucose()`, `deleteGlucose()`
- **Insulin:** `loadInsulinAudit()`, `editInsulin()`, `deleteInsulin()`
- **Intake:** `loadIntakeAudit()`, `editIntake()`, `cancelIntakeEdit()`, `deleteIntake()`
- **Supplement intake:** `loadSupplementIntakeAudit()`, `deleteSupplementIntake()`
- **Events:** `loadEventAudit()`, `deleteEvent()`
- **Nutrition master:** `loadNutritionAudit()`, `deleteNutritionItem()`
- **Supplements master:** `editSupplementRow()`, `saveSupplementRow()`, `deleteSupplement()`

### main.js
- `initializeApp()` — initialises all modules and loads initial data
- Entry point; runs on `DOMContentLoaded`

## Load Order

Script dependency order (defined in `index.html.dev`; respected by `build-js.py`):

1. `config.js` — no dependencies
2. `utils.js` — depends on config.js
3. `tabs.js` — depends on utils.js
4. `dashboard.js` — depends on config.js, utils.js
5. `data-loader.js` — depends on config.js, dynamic-items.js
6. `dynamic-items.js` — depends on data-loader.js
7. `forms.js` — depends on utils.js, data-loader.js
8. `audit.js` — depends on config.js, utils.js, data-loader.js
9. `main.js` — depends on all modules

## Maintenance Guidelines

1. **Adding a new feature** — add logic to the appropriate module; run `./build-js.py` before deploying
2. **API endpoint changes** — update `config.js` only
3. **Styling** — all CSS in `css/styles.css`
4. **New data types** — loading functions go in `data-loader.js`
5. **New forms** — handlers go in `forms.js`; use `setButtonLoading()` / `resetButton()`
6. **New audit views** — functions go in `audit.js`; use `formatTimestamp()` for display
7. **New dashboard charts** — render functions go in `dashboard.js`; append `&tz=` to API calls
8. **Script order change** — edit `index.html.dev` only, then run `./build-js.py`

## Testing

# Production (build first)
./build-js.py
MTLS_ENABLED=true PORT=8443 python3 server.py

python3 test_server.py
